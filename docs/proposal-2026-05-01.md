## 1. 재고관리 시스템 분석 및 개선안

### 1.1 현재 상태 진단

| 기능 | 상태 | 비고 |
|------|------|------|
| 재고 조회/필터링 | **정상** | 페이지네이션, 카테고리, 검색, 안전재고 필터 |
| 입고 처리 | **정상** | 채번, 수량 반영, 거래 이력 기록 |
| 출고 처리 | **정상** | 재고 차감, 거래 이력 기록 |
| 재고 조정 | **정상** | +/- 조정, 사유 기록 |
| 재고 통계 | **정상** | 요약 카드, 안전재고 알림 |
| 입고 검수 | **절반** | API 있음, **UI 미구현** |
| 재고 실사 | **고장** | 품목 조회 버그 (아래 상세) |
| 자동 차감 | **미연결** | 테이블 존재, UI 미통합 |
| 창고 위치 관리 | **미연결** | `/storage-zones` 페이지 있으나 재고와 미연결 |

### 1.2 즉시 수정 필요 (버그)

**재고 실사 품목 로딩 실패**
- `src/routes/inventoryCount.ts` 72행: `WHERE i.category_name = 'purchase_item'` (잘못된 조건)
- 올바른 조건: `WHERE i.is_purchase_item = 1`
- 영향: 재고 실사 생성 시 품목이 0건으로 나옴 → **기능 사용 불가**

### 1.3 제거 검토 대상

| 항목 | 이유 | 판단 |
|------|------|------|
| `inventory_items` 테이블 | 0134 마이그레이션에서 FK를 `items`로 이전 완료. 레거시 잔재 | **제거 가능** |
| `productionReports.ts` 369행 `inventory_items` JOIN | 빈 테이블 조인 → 항상 0건 반환 | **수정 필요** |
| 입고 시 즉시 재고 반영 | 검수(inspection) 워크플로우와 충돌 — 불합격이어도 이미 재고 증가 | **설계 재검토** |

## 2. 현장 오퍼레이터 작업지시 개선안

### 2.1 현재 기능 실사 (코드 기반 확인)

| 기능 | 상태 | 상세 |
|------|------|------|
| 긴급도 색상 코딩 | **정상** | D-0(빨강) ~ D-4+(초록) 좌측 보더 |
| 배송 카운트다운 | **정상** | "D-1 14:30", "마감!" 등 |
| 칸반 보드 | **존재** | 2컬럼(진행중/출력완료) + 보류 섹션, 드래그앤드롭 지원 → **동작 검증 필요** |
| QR 스캔 입력 | **제한적** | 숨겨진 input으로 카드번호 스캔 → 출고 처리만 지원. **QR 코드 자체 생성은 없음** |
| 진행률 표시 | **존재** | `print_progress` 퍼센트 바 + "3/5" 텍스트 → 칸반 카드에 있으나 **눈에 띄지 않음** |
| 후가공 뱃지 | **존재** | 회색 필 뱃지로 PP 이름 표시. 마진/상세 정보 없음 |
| 마감방식 | **미표시** | DB에 finishing JSON 저장되지만 카드 UI에 미반영 |
| 고객 메모 | **제한적** | `card.notes` 필드가 상세 모달에만 표시. 주문서 특이사항은 미연동 |
| 작업지시서 출력 | **없음** | 인쇄 가능한 작업 카드/PDF 없음 |

### 2.2 개선 항목 (검토 완료)

#### A. 후가공 카드 표시 분리 + 마감방식 통합 (구현 완료)

**구현된 기능: `display_on_card` DB 플래그**
- `post_processing_options` 테이블에 `display_on_card INTEGER DEFAULT 1` 컬럼 추가 (마이그레이션 0177)
- 후가공 관리 UI에서 눈 아이콘 토글로 변경 가능
- `cards.js`에서 `display_on_card=0` 항목 자동 필터링

| 항목 | display_on_card | 사유 |
|------|----------------|------|
| 코팅(라미네이팅) | 1 (표시) | 작업자 수동 코팅 |
| 펀칭 | 1 (표시) | 작업자 수동 타공 |
| 마감방식 | 1 (표시) | 작업자 수동 미싱/재단 |
| 주석 | 0 (숨김) | IA 자동 생성 |
| 오프셋(다이컷) | 0 (숨김) | IA 자동 처리 |
| 돔보 | 0 (숨김) | IA 자동 생성 |

**칸반 카드에 표시할 내용 (작업자 관련만):**
```
┌ 후가공 ──────────────────────┐
│ [코팅] [타공:4모]            │
│ 마감: 접어미싱(4cm) 사방     │
└──────────────────────────────┘
```

- 후가공: `display_on_card=1`인 항목만 뱃지 표시 + 마진 값 괄호 표기
- 마감방식: 사방 동일이면 `접어미싱(4cm) 사방`, 다르면 `상:접어미싱 하:열재단 좌:줄미싱 우:줄미싱`
- 구현: `cards.js`의 `buildKanbanCard()`와 `buildGridCard()`에서 `card.finishing` JSON 파싱 추가 (착수 가능)
- **순서/작업 설명은 추후 업데이트 예정**

#### B. 진행률 가시성 강화 (단기, 착수 가능)

현재 `print_progress` 퍼센트 바가 존재하지만 칸반 카드에서 눈에 안 띔.

**제안: 카드 상단 우측에 원형 뱃지로 표시**
```
┌──────────────────── [3/5] ──┐  ← 원형 뱃지 (미완료: 주황, 완료: 초록)
│ (주)ABC광고                  │
│ 일반현수막 100×200cm ...     │
```

- 위치: 카드 헤더 영역 우상단 (긴급도 뱃지 옆)
- 스타일: 
  - 미완료: `bg-amber-100 text-amber-700` 원형 뱃지 `3/5`
  - 완료: `bg-green-100 text-green-700` 원형 뱃지 `5/5 ✓`
  - 단건: 표시 생략 (1/1은 불필요)
- 구현: `buildKanbanCard()` 상단에 조건부 뱃지 추가

#### C. 주문서/고객 특이사항 연동 (단기~중기)

현재 `card.notes`만 상세 모달에 표시. 주문서의 `internal_notes`와 거래처 `client_notes`가 오퍼레이터에게 도달하지 않음.

**제안:**
- 카드 상세 패널에 **"주문 메모"** 섹션 추가 → `orders.internal_notes` 표시
- 카드 상세 패널에 **"거래처 메모"** 섹션 추가 → `client_notes` 최근 3건 표시
- 칸반 카드에는 메모가 있으면 📝 아이콘만 표시 (공간 절약)
- 구현: cards API의 GET /:id에서 JOIN으로 `orders.internal_notes`, `client_notes` 포함

#### D. 작업지시서 (중기, 검토 중)

주문 단위로 인쇄 가능한 작업지시서 생성. QR 코드 포함.

**구조:**
```
┌─────────────────────────────────────────┐
│ [QR → /cards?order=48]  작업지시서      │
│                                         │
│ 주문번호: 20260501-001                  │
│ 거래처: (주)ABC광고                     │
│ 납기: 2026-05-05 (배송)                 │
│ 특이사항: 색상 진하게 요청              │
├─────────────────────────────────────────┤
│ #01  일반현수막 100×200cm  2EA          │
│      마감: 접어미싱(4cm) 사방            │
│      후가공: 코팅, 타공(4모)             │
│                                         │
│ #02  메쉬현수막 300×100cm  1EA          │
│      마감: 열재단 사방                   │
│      후가공: 없음                        │
└─────────────────────────────────────────┘
```

- QR: `qrcode.js` 라이브러리로 클라이언트 생성 (CDN, ~5KB)
- QR 대상 URL: `http://192.168.0.94:3000/cards?order={orderId}` → 해당 주문 카드 필터
- 인쇄: CSS `@media print` + A5 크기 최적화, `window.print()`
- 위치: 주문 상세 페이지 또는 카드 목록에 "작업지시서 인쇄" 버튼

#### E. 현장 카드(/cards) 강화 방향 (중기~장기)

현장 카드 = `/cards` 페이지가 오퍼레이터의 주 작업 화면. 별도 심플 뷰보다 **이 페이지 자체를 강화**하는 것이 효율적.

**강화 방향:**
1. 위 A~D 항목 적용으로 정보 밀도 향상
2. 칸반 뷰 안정성 검증 (드래그앤드롭, 상태 전환 정상 동작 확인)
3. QR 스캔 범위 확장: 현재 출고만 → 상태 전환(출력완료, 보류 등)도 지원
4. 모바일 터치 최적화: 버튼 크기, 스와이프 제스처

#### 기각/보류 항목

| 항목 | 판단 | 사유 |
|------|------|------|
| ~~장비 대기열/ETA~~ | **기각** | LogWatcher가 실시간 폴링은 하지만 장비별 대기열/예상 시간 데이터를 제공하지 못함. 장비 자체가 해당 API를 지원하지 않음 |
| ~~인쇄 이력/실패 기록~~ | **대체** | 매번 다른 파일/거래처이므로 이력보다 **주문 특이사항**이 더 실용적 → C항목으로 대체 |
| ~~공정별 타임라인~~ | **보류** | 용준님 검토 중 |
| ~~모바일 오프라인~~ | **기각** | 현장에서 인터넷 연결 필수 (인쇄 작업 자체가 네트워크 의존) |

## 3. 포장 라벨/스티커 시스템 제안

### 3.1 업계 현황 (리서치 요약)

주문제작 업체에서 활용하는 라벨 4종류:
- **배송 라벨**: 수령인, 운송장번호, 바코드 (택배사 연동)
- **제품 라벨**: 주문번호, 품목명, 규격, 수량 (현수막은 말아서 밴드에 부착)
- **패킹 슬립**: 박스 동봉 명세서 (주문 항목, 수량, 특이사항)
- **공정 추적 라벨**: 생산 중 부착, 어떤 주문의 어떤 공정인지 추적

### 3.2 기술 구현 방안

#### 방안 A: 브라우저 인쇄 (가장 단순, 추가 장비 불필요)
- CSS `@media print` + `@page` 설정으로 라벨 크기 맞춤
- `window.print()` 호출로 일반 프린터에서 출력
- 장점: 즉시 구현 가능, 추가 비용 없음
- 단점: 라벨 프린터 호환성 한계, 정밀한 위치 조정 어려움

#### 방안 B: 열전사 라벨 프린터 연동 (Brother/Zebra)
- **네트워크 직접 전송**: JS에서 HTTP POST로 ZPL 명령 전송 (드라이버 불필요)
- **QZ Tray** (오픈소스): WebSocket 브릿지로 브라우저 → 로컬 프린터
- 장점: 정밀한 바코드/QR, 빠른 출력, 내구성 라벨
- 단점: 라벨 프린터 구매 필요 (Brother QL-820NWB ~20만원)

#### 방안 C: 주문/카드 라벨 페이지 (권장 1단계)
- MES에 `/labels` 페이지 추가
- 주문서 또는 카드 선택 → 라벨 미리보기 → 인쇄
- 라벨 내용:
  ```
  ┌─────────────────────────────────┐
  │ [QR: mes.url/cards/123]         │
  │                                 │
  │ 20260501-001-01                 │
  │ (주)ABC광고 / 일반현수막        │
  │ 100 × 200cm  /  2EA            │
  │ 접어미싱(사방) / 주석(하)       │
  │                                 │
  │ 납기: 05/05 (D-4) 배송         │
  │ ████████████████ (바코드)       │
  └─────────────────────────────────┘
  ```

### 3.3 구현 로드맵

| 단계 | 내용 | 예상 기간 |
|------|------|-----------|
| **1단계** | 카드/주문 라벨 페이지 (브라우저 인쇄) | 2~3일 |
| **2단계** | QR 코드 생성 (qrcode.js 라이브러리) | 1일 |
| **3단계** | 패킹 슬립 (주문 단위 명세서) | 1~2일 |
| **4단계** | 열전사 프린터 연동 (필요 시) | 3~5일 |

### 3.4 현수막 업체 특화 고려사항

- 제품이 크고 비정형 → **말아서 묶은 밴드에 라벨 부착**이 실용적
- 여러 주문이 한 차량에 실릴 때 → **배송별 그룹 라벨**로 분류
- 설치 현장 배송 시 → **설치 위치/담당자 정보** 라벨에 포함
- 재주문 고객 → **이전 주문 참조 QR** 로 빠른 확인
