# 문서 동기화 및 정합성 검증

프로젝트의 모든 핵심 문서를 실제 코드 상태와 대조하여 불일치를 수정하고, 이번 세션에서 발생한 변경사항이 빠짐없이 기록되었는지 확인하는 스킬.

> 문서가 코드와 어긋나면 다음 세션에서 잘못된 판단을 하게 된다. 이 스킬은 그 간극을 매 세션 끝에 메워주는 안전망이다.

---

## 실행 워크플로우

아래 5개 검증을 병렬 에이전트(haiku)로 동시 실행한 뒤, 결과를 모아서 수정한다.

### Phase 1: 병렬 수집 (에이전트 5개)

#### Agent 1 — 프로젝트 규모 검증
실제 파일 수를 세고 `CLAUDE.md`의 "프로젝트 규모" 테이블과 비교한다.

```bash
# 각각 실행하여 실제 숫자 확인
ls src/routes/*.ts | wc -l          # API 라우터
find src/pages -name "*.ts" | wc -l  # 페이지 (포털 포함)
ls src/scripts/*.js | wc -l          # 스크립트
ls migrations/*.sql | wc -l          # 마이그레이션
ls seed_*.sql 2>/dev/null | wc -l    # Seed 파일
```

비교 대상:
- `CLAUDE.md` → "프로젝트 규모" 테이블의 숫자
- 날짜 기준("2026-XX-XX 기준")도 오늘로 갱신

#### Agent 2 — 라우트 등록 검증
`src/index.tsx`의 import/route 등록과 실제 파일을 대조한다.

확인 항목:
- `src/routes/`에 파일이 있지만 index.tsx에 import되지 않은 라우터
- `src/pages/`에 파일이 있지만 index.tsx에 app.get 등록이 없는 페이지
- index.tsx에 import하지만 실제 파일이 없는 경우 (삭제됐는데 import만 남은 경우)

#### Agent 3 — PROJECT_STATUS.md 검증
현재 "🔴 진행 중" 항목이 실제로 진행 중인지, 이번 세션에서 완료된 작업이 "🟢 완료"에 기록되었는지 확인한다.

확인 항목:
- 진행 중 항목: 차단 요소가 해결되었으면 완료로 이동해야 하는지
- 이번 세션에서 수정한 파일 목록(`git diff --stat` 또는 변경 내역)과 완료 섹션 대조
- 최종 업데이트 날짜가 오늘인지
- 🗑️ 정리 필요 섹션에 삭제 대기 항목이 실제로 존재하는지

#### Agent 4 — MEMORY.md 교훈 검증
이번 세션에서 발생한 실수, 반복 패턴, 사용자 수정 요청이 MEMORY.md에 기록되었는지 확인한다.

기록 대상 판별 기준:
- 사용자가 "아니야", "그게 아니라", "다시" 등으로 수정을 요청한 경우
- 빌드 에러, 잘못된 파라미터, 파일 잘림 등 기술적 실수가 발생한 경우
- 같은 패턴의 실수가 이전에도 있었다면 중복 기록하지 않음
- 일회성 대화, 개인정보, 비밀번호는 기록하지 않음

#### Agent 5 — 설계 문서 + SKILL.md 정합성
design-decisions.md와 관련 SKILL.md 파일이 실제 코드와 일치하는지 확인한다.

확인 항목:
- `design-decisions.md`에 이번 세션에서 내린 설계 결정이 반영되었는지
- 변경된 디자인 토큰/CSS 값이 `mes-ui-consistency/SKILL.md`와 일치하는지
- SKILL.md의 코드 예시가 실제 layout.ts/코드와 어긋나지 않는지
- 다른 스킬의 SKILL.md에서 참조하는 파일 경로가 유효한지

### Phase 2: 수정 적용

에이전트 결과를 모아서 아래 규칙에 따라 수정한다.

#### 자동 수정 (확인 없이)
- 숫자 불일치 (파일 수, 라우터 수 등) → 실제 값으로 갱신
- PROJECT_STATUS.md 최종 업데이트 날짜 → 오늘로 갱신
- CLAUDE.md 프로젝트 규모 날짜 → 오늘로 갱신

#### 사용자 확인 후 수정
- PROJECT_STATUS.md의 항목 이동 (진행중→완료, 대기→진행중)
- MEMORY.md 교훈 추가 (추가할 내용을 보여주고 확인)
- design-decisions.md 새 섹션 추가
- SKILL.md 내용 변경

#### 수정하지 않는 것
- CLAUDE.md에 코드 값(enum, 상태 목록 등) 복사 — 단일 소스 원칙 위반
- 이미 최신 상태인 문서를 불필요하게 터치

수정 완료 후 아래 형식으로 요약한다.

```

## ✅ 자동 수정
| 문서 | 항목 | 이전 | 이후 |
|------|------|------|------|
| CLAUDE.md | API 라우터 수 | 43개 | 46개 |
| CLAUDE.md | 규모 기준일 | 2026-03-30 | 2026-04-04 |

## 📝 사용자 확인 후 수정
| 문서 | 변경 내용 | 상태 |
|------|----------|------|
| PROJECT_STATUS.md | UI 개선 작업 완료 기록 추가 | ✅ 반영 |
| MEMORY.md | 에이전트 파일 잘림 교훈 추가 | ✅ 반영 |

## ⚠️ 확인 필요
- (수동 확인이 필요한 항목이 있으면 여기에)

## 🔍 정합성
- SKILL.md ↔ layout.ts: 일치
- index.tsx 라우트 등록: 누락 없음
- design-decisions.md: 최신 반영됨
```

## 에이전트 모델 배정

| 에이전트 | 모델 | 이유 |
|---------|------|------|
| Agent 1 (규모 검증) | haiku | 단순 카운트 + 비교 |
| Agent 2 (라우트 검증) | haiku | 파일 목록 대조 |
| Agent 3 (STATUS 검증) | haiku | 텍스트 비교 |
| Agent 4 (MEMORY 검증) | haiku | 대화 맥락 판단이 약간 필요하지만 haiku로 충분 |
| Agent 5 (설계+SKILL) | sonnet | CSS 값 대조 등 코드 이해 필요 |

## 언제 실행하면 좋은가

- 세션에서 코드 수정 작업을 마친 후 ("정리해줘", "리뷰해줘")
- `npm run build` 성공 후 커밋/배포 전
- 세션 마무리 시 (다음 세션의 맥락 보존)
- 구조적 변경 후 (라우트 추가/삭제, 페이지 추가/삭제, 마이그레이션 추가)
- 사용자가 명시적으로 "문서 동기화", "/sync-docs" 요청 시
